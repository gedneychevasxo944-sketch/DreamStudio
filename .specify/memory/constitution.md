# DreamStudio 项目章程

## 核心原则

### 一、方案先行
所有代码修改前，必须先出方案让用户 review。

### 二、清晰沟通
先理解需求，提问题确认，不要猜测。

### 三、小步快走
每次只改一个功能点，验证后再继续。

### 四、追踪影响
改完后说明影响了哪些文件、哪些功能。

## 工作流程

```
用户提出需求
    ↓
分析问题，明确需求（必要时提问）
    ↓
出方案（涉及代码改动的必须出方案）
    ↓
用户 review 方案
    ↓
确认后开发
    ↓
验证功能，说明改动范围
```

## 常见坑和注意事项

### API 错误处理
- 异步操作要 try-catch
- 注意检查 nodeId 是否变化（取消请求）

### props 层层传递
- 组件 props 通过 map/对象传递，注意检查传递链路

### CSS 样式冲突
- 大量使用 CSS 类名，注意命名冲突
- 优先使用组件级别的 CSS 文件

### 状态管理
- 使用 Zustand stores
- 全局事件通过 eventBus（`on/off/emit`）或 document 自定义事件

## 提交规范

### Commit Message 格式
```
<类型>: <简短描述>

<详细说明（可选）>
```

### 类型
- `feat`: 新功能
- `fix`: 修复 bug
- `refactor`: 重构
- `ui`: UI 修改
- `chore`: 构建/工具

## 治理

### 章程修订程序
1. 提议的变更必须记录理由和影响
2. 变更需经用户 review 后方可采纳
3. 所有 PR 和变更必须验证对章程的遵循

### 合规审查
- 所有实现必须符合核心原则
- 复杂性必须有明确的项目价值支撑
